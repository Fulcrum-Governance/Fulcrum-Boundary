# Fulcrum Boundary v0.11.0

Fulcrum Boundary v0.11.0 is the verifiable-records-breadth release: a decision
record now verifies in Go, Python, TypeScript, or Rust — each an independent
re-implementation of the one `decision_hash` check, pinned to the Go
implementation by committed conformance vectors and enforced in CI. It also
adds opt-in Ed25519 signing (off by default), shell-completion and help
polish, and deeper parse/boot robustness coverage.

The release story stays narrow. Boundary decides before execution for routed
agent-tool actions and records the verdict in a hash-verifiable decision
record; v0.11.0 widens who can recompute that record's hash and lets operators
who manage keys attach authorship to it. It does not add a new governed action
surface, does not add a transport adapter, and does not upgrade any preview
surface to production.

MCP remains the first production route. Secure GitHub, Command Boundary, and
Edit Boundary remain preview surfaces.

## Install

```bash
brew install fulcrum-governance/tap/boundary
boundary selftest
boundary demo github-lethal-trifecta    # Lane 1: MCP, the first production route
boundary demo tamper-evidence           # forge the receipt, watch verification refuse it
```

The formula and the release archives install the static (`CGO_ENABLED=0`)
build: the Postgres AST classifier is unavailable, every routed SQL statement
classifies as `UNKNOWN` and is denied fail-closed, and the static build never
allows SQL the cgo build would deny. The `_cgo` archives carry the full
classifier. Building from source requires Go 1.25+ and, for the default cgo
build, a C toolchain:

```bash
go install github.com/fulcrum-governance/fulcrum-boundary/cmd/boundary@v0.11.0
```

The default first-run path uses fixture data. It makes no live calls and
performs no real mutations.

## What Changed

- **Verify a record in Go, Python, TypeScript, or Rust.** Standalone
  TypeScript (`verifiers/typescript/`) and Rust (`verifiers/rust/`) verifiers
  join the Python verifier from v0.10.0, each recomputing `decision_hash` via a
  stock RFC 8785 / JCS implementation (the `canonicalize` npm package; the
  `serde_jcs` crate) with no Boundary code on the path. Committed conformance
  vectors pin all four implementations to the same canonical bytes, enforced in
  CI. The verifiers check integrity only, not authenticity.
- **Opt-in Ed25519 signing of decision records.** Off by default. `boundary
  serve --receipt-seed FILE` (a 64-hex seed file) signs every emitted record,
  and `boundary verify-record --verify-signature --public-key <hex|file>`
  checks the signature over the recomputed `decision_hash`. `serve` startup
  fails closed (exit 1, before the listener opens) on a missing, short, or
  non-hex seed, and signature verification fails closed on a missing or
  mismatched signature. Signing never changes `decision_hash`; a signature
  attests who signed the record, not the verdict, execution, or key custody.
  The Python, TypeScript, and Rust verifiers stay integrity-only and do not
  check signatures.
- **CLI polish.** `boundary completion bash|zsh|fish` generates static shell
  completion scripts from the binary's own command table (static by design —
  regenerate after upgrades), alongside the `help` routing and `--json` verbs
  carried forward from v0.10.0.
- **Robustness depth.** Three native Go fuzz targets —
  `FuzzDecisionRecordRoundTrip`, `FuzzPolicyParse`, and `FuzzSQLClassifier` —
  run their seed corpora on every `go test ./...` and run the mutation engine
  for 60s each in a required CI `fuzz` job. A hermetic serve-boot integration
  test (`tests/serve_boot/`) asserts the governed deny (JSON-RPC `-32001`)
  before upstream on a real binary boot, and new RESP codec unit tests cover
  the kernel trust path's hand-rolled Redis protocol.
- **Adapter production bar.** `docs/ADAPTER_PRODUCTION_BAR.md` documents the
  mechanical bar — lifecycle steps, bypass-proof delegation, fail-closed
  transports, on-disk test evidence — and the process for advancing an adapter
  from `preview` to `production`, with a worked `mcp`-vs-`webhook` example.

## Verify In Your Stack

Export a record once, then recompute its `decision_hash` with whichever
toolchain you trust — none of them link Boundary code:

```bash
boundary demo github-lethal-trifecta --out lane1.txt
RECORD=github-lethal-trifecta-artifacts/decision-record.json

# Go (the binary)
boundary verify-record "$RECORD"

# Python — stock rfc8785
pip install rfc8785
python3 verifiers/python/boundary_verify.py "$RECORD"

# TypeScript — stock canonicalize (JCS)
( cd verifiers/typescript && npm install && \
  node --experimental-strip-types boundary_verify.ts "../../$RECORD" )

# Rust — stock serde_jcs
cargo run --manifest-path verifiers/rust/Cargo.toml -- "$RECORD"
```

All four recompute the same `decision_hash` from the record bytes — committed
conformance vectors pin the non-Go verifiers to the Go implementation, enforced
in CI. The hashes are unkeyed SHA-256 over RFC 8785 canonical bytes: integrity,
not authenticity. A passing check detects post-emission tampering; it does not
prove who produced the record, that the action was executed or prevented, or
that the verdict was correct.

## Signing, And What It Establishes

Signing is opt-in and off by default; with no seed configured, records are
unsigned and byte-identical to the unsigned path. A valid Ed25519 signature
proves the record was signed by the holder of the named key, over exactly the
content `decision_hash` covers — it does not prove the verdict was correct,
does not prove the governed action executed or was prevented, and does not
solve key custody, which is the operator's responsibility. Signing never
changes `decision_hash`, so a signed record and its unsigned twin hash
identically, and the Python, TypeScript, and Rust verifiers — integrity-only —
do not check signatures; signature checking is the Go `verify-record
--verify-signature` path alone. See [`SIGNING.md`](../SIGNING.md) for key
generation, the failing-closed behavior, and the `#143` scope note that
`mcp proxy` / install-time enablement is not yet wired.

```bash
boundary serve --policies ./policies/ --receipt-seed ./boundary-receipt.seed
boundary verify-record --verify-signature --public-key boundary-receipt.pub record.json
```

## Two Proof Lanes

| Lane | Status | Demo | What is denied | Verified shape |
| --- | --- | --- | --- | --- |
| MCP — the first production route | production | `boundary demo github-lethal-trifecta` | A write-after-taint GitHub action, denied before upstream | `actual=DENY`, `upstream_called=false`, `reason=lethal_trifecta_detected` |
| Command Boundary — a delivered preview (routed-only) | delivered preview | `boundary demo command-secret-exfil` | A routed `curl -d @.env ...` secret exfiltration, denied before execution | `actual=DENY`, `executed=false`, `class=C6` |

## What It Does Not Prove

| Limit | Why it matters |
| --- | --- |
| Universal attack prevention | Fixture demos and policy tests cover supplied paths only. |
| Production Secure GitHub | Secure GitHub remains preview until deployment bypass evidence and broader live coverage exist. |
| Production Command Boundary | Command Boundary remains routed-path-only preview; direct shell access is not governed. |
| Production Edit Boundary | Edit Boundary remains routed-edit-envelope-only preview. |
| Deployment bypass resistance | A passing demo or policy test does not prove every direct or unrouted path to the same tool has been removed. |
| Blanket standards conformance | The RFC 8785 statement is scoped to the decision record's canonical bytes; it is not a whole-product conformance claim. |
| Cryptographic record authenticity | Signing is opt-in and off by default; a signature attests the signer, not the verdict, execution, or key custody. |
| Cross-language signature verification | The Python, TypeScript, and Rust verifiers are integrity-only; they recompute `decision_hash` and do not check signatures. |
| Static-build SQL classification | The static build classifies routed SQL as `UNKNOWN` and denies it fail-closed; it does not classify statements the cgo build would. |
| Hook coverage | The Claude Code hook governs only hook-delivered tool calls; it does not govern direct shell or editor access outside the hook. |
| Topology attestation | `topology_profile` records an asserted posture; it does not verify the deployment matches it. |
| Independent no-upstream proof | `execution_claim` is an adapter self-report; `boundary replay` reproduces the decision, not the absence of upstream side effects. |
