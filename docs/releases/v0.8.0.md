# Fulcrum Boundary v0.8.0

Fulcrum Boundary v0.8.0 is the first release to include the Phase 0A "Trust the
Record" lane: route-context decision records (`DecisionRecordV2`),
`boundary explain`, `boundary replay`, and uniform record-location output. These
features were already on `main`; v0.8.0 is the tag that puts them in the
`@v0.8.0` install. It adds no new governed action surface and upgrades no preview
surface to production.

MCP remains the first production route. Secure GitHub, Command Boundary, and
Edit Boundary remain preview surfaces.

## Install

Requires Go 1.25+ and a C toolchain (a C compiler such as gcc/clang on `PATH`).
The default build links the Postgres SQL classifier (`pganalyze/pg_query_go`)
via cgo, so `CGO_ENABLED=0` builds fail.

```bash
go install github.com/fulcrum-governance/fulcrum-boundary/cmd/boundary@v0.8.0
boundary selftest
boundary demo github-lethal-trifecta    # Lane 1: MCP, the first production route
boundary demo command-secret-exfil      # Lane 2: Command Boundary, a delivered preview
```

The default first-run path uses fixture data. It makes no live calls and
performs no real mutations.

## What Changed

- Route-context decision records (`DecisionRecordV2`, `schema_version "2"`): an
  additive superset over `DecisionRecordV1` carrying `adapter_id`, `route_id`,
  `topology_profile`, and `execution_claim`. V1 (`schema_version "1"`) records
  stay valid and byte-compatible; the route-context fields are content covered by
  `decision_hash`. `topology_profile` is asserted, not attested; `execution_claim`
  is an adapter self-report and is not independently corroborated by the hashed
  record.
- `boundary explain` renders a decision record (V1 or V2) read-only — verdict,
  reason, decision mode, matched rule, route context, and what each of the three
  hashes covers — with a stable `--format json` (`boundary.explain.v1`). It does
  not re-verify hashes or prove the verdict was correct or enforced.
- `boundary replay` re-evaluates a recorded request against the recorded policy
  bundle and reproduces the recorded decision-defining fields, failing closed on
  any mismatch. It reproduces the decision, not enforcement and not the absence of
  upstream side effects; it is routed-only.
- `boundary verify-record` now accepts `schema_version` of `"1"` or `"2"` and
  recomputes `decision_hash` per the record's own version.
- Uniform record-location output across the record-emitting commands (`demo`,
  `redteam`, `evidence`): a single stable record-path/`record_id` line and uniform
  `--out` semantics, so the find → verify → explain → replay loop is copy-paste.
- Docs hardening carried forward: `docs/TROUBLESHOOTING.md`,
  `docs/ROUTE_CONFORMANCE_CHECKLIST.md`, and the committed `docs/examples/`
  walkthrough.
- Bumped active install and GitHub Action examples to `@v0.8.0` and set
  `CITATION.cff` to `0.8.0`.

## The Record-Trust Loop

The Phase 0A commands compose into a local, copy-paste loop over a record a
verdict leaves behind:

```bash
boundary demo github-lethal-trifecta --json --out demo.json   # emits a record at a documented path
boundary verify-record <record>                               # recomputes the three hashes
boundary explain <record>                                     # renders the verdict + route context
boundary replay <record> --request <req> --policies <dir>     # reproduces the decision
```

This loop reproduces and renders the recorded *decision*. It does not prove
enforcement, that the deployment matches the recorded `topology_profile`, or that
no upstream bytes moved.

## Two Proof Lanes

| Lane | Status | Demo | What is denied | Verified shape |
| --- | --- | --- | --- | --- |
| MCP — the first production route | production | `boundary demo github-lethal-trifecta` | A write-after-taint GitHub action, denied before upstream | `actual=DENY`, `upstream_called=false`, `reason=lethal_trifecta_detected` |
| Command Boundary — a delivered preview (routed-only) | delivered preview | `boundary demo command-secret-exfil` | A routed `curl -d @.env …` secret exfiltration, denied before execution | `actual=DENY`, `executed=false`, `class=C6` |

## What It Does Not Prove

| Limit | Why it matters |
| --- | --- |
| Universal attack prevention | Fixture demos cover the tested paths only. |
| Production Secure GitHub | Secure GitHub remains preview until deployment bypass evidence and broader live coverage exist. |
| Production Command Boundary | Command Boundary remains routed-path-only preview; direct shell access is not governed. |
| Production Edit Boundary | Edit Boundary remains routed-edit-envelope-only preview. |
| Cryptographic release provenance | `boundary version` reports metadata; it is not a signing or attestation system. |
| Topology attestation | `topology_profile` records an asserted posture; it does not verify the deployment matches it. |
| Independent no-upstream proof | `execution_claim` is an adapter self-report; `boundary replay` reproduces the decision, not the absence of upstream side effects or enforcement. |
| Cryptographic record authenticity | The record hashes prove integrity, not authenticity; Boundary does not emit `proved` decisions. |
