# Fulcrum Boundary v0.6.0

Fulcrum Boundary v0.6.0 packages the Filesystem/Edit Boundary preview. It keeps
the public release wedge honest: MCP remains production; Secure GitHub, Command
Boundary, and Edit Boundary remain preview; and the default first-run path
stays credential-free and mutation-free.

## Install

Requires Go 1.25+.

```bash
go install github.com/fulcrum-governance/fulcrum-boundary/cmd/boundary@v0.6.0
boundary selftest
boundary demo github-lethal-trifecta
```

The demo uses fixture data. It makes no live GitHub calls and performs no real
mutations.

## What Shipped

- Filesystem/Edit Boundary preview for proposed file mutations routed through
  Boundary edit envelopes.
- `boundary edit inspect` for classifying patch bytes without applying them.
- `boundary edit apply` for evaluating a proposed edit through the shared
  governance pipeline before applying it.
- Edit decision records with patch hash, class, action, file list, and applied
  state.
- Fixture-only Edit Boundary redteam packs for secret-bearing, package-script,
  CI/deploy, destructive-delete, and cross-scope mutation paths.
- Public docs, docs-site pages, claims ledger, release truth, install docs, and
  GitHub Action examples aligned to `@v0.6.0`.

## Try The Preview Locally

```bash
boundary edit inspect --patch proposed.diff
boundary edit apply --patch proposed.diff --dry-run
boundary redteam --pack edit-secret-exfil
```

Edit Boundary is preview. It governs only proposed file mutations represented
as Boundary edit envelopes.

## What It Proves

| Surface | Proof |
| --- | --- |
| Inspect | Patch bytes can be classified without applying them. |
| Apply wrapper | Routed edit envelopes are evaluated before writes occur. |
| Denial | Denied edit envelopes do not apply. |
| Records | Edit decisions include patch hash, class, action, file list, and applied state. |
| Redteam | Fixture edit-risk packs exercise selected unsafe mutation paths without live project mutation. |

## What It Does Not Prove

| Limit | Why it matters |
| --- | --- |
| Direct editor-write protection | Editor saves are outside Boundary unless the editor routes through an edit envelope. |
| Direct filesystem-write protection | Arbitrary process writes remain outside Boundary. |
| Shell redirection control | `>` and similar shell writes are command/shell behavior, not Edit Boundary coverage. |
| Direct `git apply` control | Direct patch application bypasses Boundary unless routed through `boundary edit apply`. |
| Filesystem sandboxing | Edit Boundary is a policy gate, not an OS sandbox. |
| Production edit governance | Production status requires deployment evidence that edit proposals route through Boundary. |

## Maturity Table

| Adapter/Profile/Surface | Status | v0.6.0 truth |
| --- | --- | --- |
| MCP | production | Production JSON-RPC MCP proxy path remains unchanged. |
| Secure GitHub | preview | Fixture proof plus opt-in live GitHub App read and denied-write conformance harness remains unchanged from v0.5.0. |
| Command Boundary | preview | Routed project-local command governance remains unchanged from v0.4.0. |
| Edit Boundary | preview | Routed edit-envelope inspection, apply gating, decision records, and fixture redteam packs. |
| CLI, CodeExec, gRPC, Managed Agents, Webhook, A2A | preview | No maturity upgrade in v0.6.0. |

## Claims Boundary

Use:

> Boundary provides preview Edit Boundary governance for proposed file
> mutations routed through Boundary edit envelopes.

Do not claim:

- Boundary controls all file writes.
- Boundary protects direct editor writes.
- Boundary prevents every unsafe edit.
- Boundary provides filesystem sandboxing.
- Boundary provides universal coding-agent file safety.
- Boundary provides production edit governance.
