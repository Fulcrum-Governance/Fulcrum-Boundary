# Fulcrum Boundary v0.5.0

Fulcrum Boundary v0.5.0 packages the Secure GitHub live conformance preview.
It keeps the public release wedge unchanged: MCP remains production, Secure
GitHub remains preview, and the default first-run path stays credential-free
and mutation-free.

## Install

Requires Go 1.25+.

```bash
go install github.com/fulcrum-governance/fulcrum-boundary/cmd/boundary@v0.5.0
boundary selftest
boundary demo github-lethal-trifecta
```

The demo uses fixture data. It makes no live GitHub calls and performs no real
mutations.

## What Shipped

- Secure GitHub GitHub App authentication support for live conformance.
- Opt-in live read conformance for operator-owned GitHub App credentials.
- Denied write-after-taint no-mutation proof for the configured live
  conformance path.
- Sanitized transcript evidence that records hashes and booleans instead of raw
  issue content or credentials.
- `boundary secure github conformance read`, `denied-write`, and `all`
  commands.
- Release truth, claims ledger, install docs, docs-site copy, and GitHub Action
  examples aligned to `@v0.5.0`.

## Opt-In Live Conformance

Live conformance is off by default. It requires operator-owned GitHub App
credentials and `BOUNDARY_GITHUB_CONFORMANCE=true`.

```bash
BOUNDARY_GITHUB_CONFORMANCE=true \
  boundary secure github conformance denied-write
```

The denied-write conformance path must report:

```text
actual action: DENY
reason: lethal_trifecta_detected
upstream_called=false
github_mutation_called=false
```

Missing required GitHub App environment fails closed when live conformance is
enabled. Without opt-in, conformance commands skip without network calls.

## What It Proves

| Surface | Proof |
| --- | --- |
| Fixture demo | The tested GitHub write-after-taint fixture is denied before upstream mutation. |
| Live read conformance | A configured real issue can be read through GitHub App installation auth and recorded as sanitized taint evidence. |
| Live denied-write conformance | A protected write-after-taint action is denied before any GitHub mutation client call executes. |
| Transcript safety | Evidence records hashes, booleans, and sanitized metadata instead of raw secrets or raw issue content. |

## What It Does Not Prove

| Limit | Why it matters |
| --- | --- |
| Production Secure GitHub | Secure GitHub remains preview until deployment bypass evidence and broader live coverage exist. |
| Deployment bypass resistance | Direct GitHub API, upstream GitHub MCP, `gh`, `git`, browser, CI, SSH, and other credentialed paths remain outside Secure GitHub unless deployment topology removes or routes them. |
| Universal prompt-injection defense | The harness covers the configured read-taint and denied-write path, not every malicious issue or agent behavior. |
| Full GitHub MCP catalog coverage | v0.5.0 covers the live conformance path, not every GitHub tool. |

## Claims Boundary

Use:

> Secure GitHub can use real GitHub App credentials to read real GitHub
> context, mark the session tainted, and deny a protected write-after-taint
> action before any upstream GitHub mutation client call executes.

Do not claim:

- Secure GitHub is production.
- "Boundary fully secures GitHub" — not a permitted claim; only the routed
  read-taint and denied-write conformance path is governed.
- Live conformance proves deployment bypass resistance.
- Boundary mutates live repositories during conformance by default.
- Boundary covers the full GitHub MCP catalog.
