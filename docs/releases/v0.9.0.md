# Fulcrum Boundary v0.9.0

Fulcrum Boundary v0.9.0 is the developer-trust release: it keeps the Phase 0A
"Trust the Record" loop from v0.8.0 and adds Phase 1 policy-as-code testing with
`boundary test`.

The release story is deliberately narrow. Boundary decides before execution for
routed agent-tool actions, records the verdict, lets developers inspect and
replay that decision, and now lets teams assert expected policy verdicts in CI.
It does not add a new governed action surface, does not add a transport adapter,
and does not upgrade any preview surface to production.

MCP remains the first production route. Secure GitHub, Command Boundary, and
Edit Boundary remain preview surfaces.

## Install

Requires Go 1.25+ and a C toolchain (a C compiler such as gcc/clang on `PATH`).
The default build links the Postgres SQL classifier (`pganalyze/pg_query_go`)
via cgo, so `CGO_ENABLED=0` builds fail.

```bash
go install github.com/fulcrum-governance/fulcrum-boundary/cmd/boundary@v0.9.0
boundary selftest
boundary demo github-lethal-trifecta    # Lane 1: MCP, the first production route
boundary demo command-secret-exfil      # Lane 2: Command Boundary, a delivered preview
```

The default first-run path uses fixture data. It makes no live calls and
performs no real mutations.

## What Changed

- `boundary test` runs operator-authored policy-as-code cases against local
  Boundary policy bundles. Each case names a local policy bundle, a routed
  request fixture, and an expected verdict. The runner exits non-zero on any
  mismatch, malformed case, unexpected policy-load error, or failed
  `parse_rejection` expectation.
- `boundary test --format json` emits a stable `boundary.test.v1` envelope with
  one result per case and explicit local-safety flags:
  `requires_credentials=false`, `requires_network=false`, and
  `mutates_live_systems=false`.
- The committed golden corpus under `tests/fixtures/policy-test/cases` covers
  `allow`, `deny`, `warn`, `require_approval`, `escalate`, and expected
  `parse_rejection`.
- The standard release gate now runs the policy-test corpus, so policy-as-code
  testing is part of release evidence rather than a docs-only claim.
- `docs/POLICY_TESTING.md` provides the canonical operator guide and CI recipe.
- Public install and GitHub Action examples now target `@v0.9.0`.
- `CITATION.cff` is set to `0.9.0`.
- Decision-record docs now distinguish the single-record `decision record path:`
  artifact from the multi-record `decision record log:` JSONL stream.

## Policy-As-Code Test Loop

```bash
boundary test --path tests/fixtures/policy-test/cases
boundary test --path tests/fixtures/policy-test/cases --format json
```

Operator-owned repositories can keep policy tests beside their policy bundles:

```bash
boundary test
boundary test --format json
```

This is a local, fixture-only assertion runner. It does not require credentials,
does not use the network, does not call upstream tools, and does not mutate live
systems. A passing run reports policy verdicts for supplied routed request
fixtures only; it does not prove production route enforcement, deployment bypass
resistance, or global verdict correctness beyond those fixtures.

## Record-Trust Loop Carried Forward

The Phase 0A commands still compose into a local, copy-paste loop over a record a
verdict leaves behind:

```bash
boundary demo github-lethal-trifecta --json --out demo.json   # emits a single-record JSON path
boundary verify-record <record>                               # recomputes the stable hashes
boundary explain <record>                                     # renders the verdict + route context
boundary replay <record> --request <req> --policies <dir>     # reproduces the decision
```

This loop reproduces and renders the recorded *decision*. It does not prove
enforcement, that the deployment matches the recorded `topology_profile`, or that
no upstream bytes moved.

## Two Proof Lanes

| Lane | Status | Demo | What is denied | Verified shape |
| --- | --- | --- | --- | --- |
| MCP — the first production route | production | `boundary demo github-lethal-trifecta` | A write-after-taint GitHub action, denied before upstream | `actual=DENY`, `upstream_called=false`, `reason=lethal_trifecta_detected` |
| Command Boundary — a delivered preview (routed-only) | delivered preview | `boundary demo command-secret-exfil` | A routed `curl -d @.env ...` secret exfiltration, denied before execution | `actual=DENY`, `executed=false`, `class=C6` |

## What It Does Not Prove

| Limit | Why it matters |
| --- | --- |
| Universal attack prevention | Fixture demos and policy tests cover supplied paths only. |
| Production Secure GitHub | Secure GitHub remains preview until deployment bypass evidence and broader live coverage exist. |
| Production Command Boundary | Command Boundary remains routed-path-only preview; direct shell access is not governed. |
| Production Edit Boundary | Edit Boundary remains routed-edit-envelope-only preview. |
| Production route enforcement from policy tests | `boundary test` evaluates routed request fixtures against local policies; deployment topology evidence is separate. |
| Deployment bypass resistance | Passing policy tests do not prove every direct or unrouted path to the same tool has been removed. |
| Cryptographic release provenance | `boundary version` reports metadata; it is not a signing or attestation system. |
| Topology attestation | `topology_profile` records an asserted posture; it does not verify the deployment matches it. |
| Independent no-upstream proof | `execution_claim` is an adapter self-report; `boundary replay` reproduces the decision, not the absence of upstream side effects or enforcement. |
| Cryptographic record authenticity | The record hashes prove integrity, not authenticity; Boundary does not emit `proved` decisions. |
